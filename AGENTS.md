# AGENTS.md

0) Non-negotiable Locality Envelope (default)
- Vanilla-first: prefer platform primitives; default to vanilla first, and only use frameworks if justified.
- Dependency budget: 0 by default; max 1-2 per concern, only if justified.
- File budget: Fewer is better. Create as fewer files as possible to achieve production-grade quality without creating context overwhelm (zero unnecessary files).
- LOC budget: target <= 1000 LOC/file; split when > 2000 LOC. Avoid context rot growth.
- Every Plan offered must explicitly state: {files, LOC/file, deps}.

1) Role routing (global)
- Keep bridgecode playbooks in `bridgecode/` and generated artifacts in `agents/`; do not mix the two folders.
- When doing backend development always follow the LLM_FRIENDLY_ENGINEERING_BACKEND instructions.
- When doing frontend design always follow the LLM_FRIENDLY_ENGINEERING_FRONTEND instructions.
- When planning or debugging always follow the LLM_FRIENDLY_PLAN_TEST_DEBUG instructions.
- Whenever starting a conversation (no previous turns detected) check only one time, to start the conversation, the `agents/canon.md` file to get an overview of the repo you're working with.
  - If `agents/canon.md` does not exist, instruct the user to use `LIRA` and the sub-role `senior`.
- Whenever `EYE` fixes an error, through simple implementation or isolated debugging (with or without loop-breaker), add a compoacted, dense one-line with the solution to the error (to be prevented) in the Specific repo rules within the `/AGENTS.md` file.
  - If a correction for an error already exist, either modify that one-line (if the correction was insufficient) or add extra instructions (if new related errors emerged but the initial correction was correct).
- Codex bridgecode routing:
  - If the user invokes `INSTRUCT`, immediately read `bridgecode/INSTRUCT.md` and follow it. When that playbook routes to negentropy behavior, read `bridgecode/negentropy.md`.
  - If the user invokes `LIRA` for a new project or architecture definition, read `bridgecode/LIRA.md` and `bridgecode/architect.md`.
  - If the user invokes `LIRA` for an existing codebase review or asks to audit/review the repo, read `bridgecode/LIRA.md` and `bridgecode/senior.md`.
  - If the user invokes `EYE`, or asks to implement/debug work using bridgecode execution, read `bridgecode/EYE.md` and `bridgecode/plan-code-debug.md`.
- `bridgecode/` contains the Codex playbooks. `agents/` is reserved for generated artifacts such as `canon.md`, `plan.md`, `review.md`, `negentropized_instructions.md`, and `agents/testscripts/failure_report.md`.

Default to language primitives and standard libraries (vanilla-first mandate) before introducing any external dependency, and when dependencies prove necessary, assign exactly one tool per architectural concern, wrapping specialized libraries in thin adapter interfaces that isolate external complexity at system edges. Keep files purposeful and focused, naming them with boring descriptive terminology, and promote code to shared locations only after the rule of three (genuine reuse appears three times) proves the pattern stable. Define explicit schema-validated contracts at every feature boundary, validating all inputs at entry and shaping all outputs at exit using uniform error models that include status, code, message, and optional details without leaking implementation specifics. Confine singletons (database clients, cache clients, auth providers, config loaders, logging systems) strictly to infrastructure zones as stateless factories, never embedding domain logic within them. Limit build operations to essential commands (install, develop, test, build, start) with pinned dependencies ensuring deterministic reproducible builds, and implement structured feature-scoped logging with request correlation for operational transparency. Write unit tests for key pure functions, end-to-end tests for critical paths, and contract tests for boundary schemas, co-locating all tests with the code they validate. Target established accessibility compliance standards with keyboard-first interaction, visible focus states, reduced motion respect, and appropriate ARIA roles, while centralizing design tokens (color, spacing, typography, radius, motion) in exactly one style system per project. Begin architecture as a simple monolith of vertical slices, extracting async work to background jobs only when genuine need arises, creating shared packages only after cross-feature reuse stabilizes, and splitting into services only when isolation, scaling, or compliance boundaries are demonstrated through measured metrics. When refactoring, inline logic toward callers before extracting abstractions, consolidate fragmented files into coherent feature folders, and write or update contracts before changing implementation—replacing unnecessary libraries with vanilla implementations and removing abstractions that no longer provide value. Reject utility collections mixing unrelated concerns, code fragmentation into many tiny files increasing navigation cost, duplicate tools serving identical purposes, implicit global state, reflection-heavy patterns obscuring behavior, and any framework with sparse documentation on critical paths. Ensure every deliverable constitutes a self-contained feature slice with adjacent schema and tests, confining global concerns to singleton adapters without domain logic, providing run instructions fitting three lines producing deterministic results, and documenting selected tools while verifying they meet documentation and predictability thresholds. Present options briefly with clear default recommendations, state assumptions explicitly and proceed without unnecessary blocking, produce small copyable immediately-runnable artifacts, prefer diagrams and checklists before implementation code, generate code in vertical slices rather than global sweeps, refactor incrementally while preserving existing contracts, and ensure every artifact reaches completeness enabling immediate utility. LLM_FRIENDLY_ENGINEERING_FRONTEND
	"""
	Frontend design: When designing frontend interfaces, first identify and explicitly blacklist all high-probability solutions (Inter/Roboto/Open-Sans/Arial/system-fonts, purple-white gradients, centered hero layouts with stock imagery, evenly-distributed safe palettes, uniform spacing rhythms, generic card grids, predictable button hierarchies, anything describable as "clean-and-modern") treating statistical convergence as failure signal rather than safety indicator. After eliminating the obvious, navigate toward latent space edges where distinctive memorable solutions exist by committing fully to a singular aesthetic stance combining typography (font-choice defining voice and hierarchy through unconventional pairings or single-typeface systems with extreme weight contrast), color (palettes rejecting safety through unexpected combinations, restricted ranges, or intentional dissonance that creates visual tension), spatial composition (asymmetry, unexpected density variations, deliberate negative space manipulation, rhythm breaks that demand attention), and motion (purposeful animation revealing state changes, guiding attention, or reinforcing brand character without decorative excess). Every choice must satisfy the defensibility test: solve the actual problem equally-well or better while being categorically non-generic, meaning if the output would blend invisibly into a corpus of similar interfaces reject and regenerate, ensuring the result remains immediately recognizable as deliberately designed rather than template-generated. Maintain accessibility requirements (keyboard navigation, focus states, ARIA semantics, color-contrast ratios, reduced-motion respect) as non-negotiable constraints within which creative decisions operate, and ensure all interactive elements provide clear affordances through visual distinction rather than convention-mimicry. Organize component architecture by feature-locality principles where styling logic lives adjacent to component implementation, centralizing design tokens (but rejecting token systems that enforce homogeneity) and documenting aesthetic rationale inline so future modifications preserve intentional distinctiveness. Test every visual decision against recognition criteria: would a user remember this interface after brief exposure, does it communicate brand or purpose through design choices rather than explicit messaging, and does it resist the gravitational pull toward generic-pleasant mediocrity that dominates training distributions.
	"""

2) Specific repo rules
- 
- When Supabase MCP auth fails with `Unauthorized` or `Transport closed`, update both `~/.codex/config.toml` and `~/.config/opencode/opencode.json` with the current `SUPABASE_ACCESS_TOKEN`; existing `mcp-server-supabase` processes can hold stale auth until restarted, even after config is corrected.
- CSV lead imports must accept fallback profile columns such as `Legacy LinkedIn URL` and must not create a batch unless at least one normalized `linkedin_url` survives parsing; imported leads should inherit both `batch_id` and `sequence_id` together.
- Normalize outreach modes at the app boundary (`connect_message` / `connect_only`) and treat `message_only` only as a legacy alias; never let a stale mode label reach `importLeads`, lead status routing, or worker spawn arguments.
- `connect_message` is the canonical worker-facing stage for connection-plus-note sends; keep `message` only as a legacy parser alias and never emit it from app spawns, status routing, or analytics labels.
- When Next dev starts returning 404s for `main-app.js`, `app-pages-internals.js`, `layout.js`, or `layout.css`, clear `apps/web/.next` and restart `next dev` on the intended port (`3000`) so the browser stops reading chunks from a stale or wrong-port build state.
- `run_all.sh` must pre-stop any existing listener on port `3000` and clear `apps/web/.next` before starting web; parallel Next dev instances (`3000` + fallback `3001`) create stale chunk/CSS mismatch that appears as a fully unstyled UI.
- Hostinger VPS public routing must go through Traefik labels on the app service; never leave `linkedin-outreach` publishing `3000` once the domain is meant to be served through the edge proxy, or the live hostname can bypass Traefik and preserve stale orphan containers.
- Hostinger auth redeploys must come from a repo-backed build context or fresh git pull, not just `updateProject`/restart on a stale VPS checkout; `createNewProject` can replace the project and clear `.env`, so verify/reseed runtime env before declaring auth fixed.
- Magic-link callbacks must use an explicit production origin env (`NEXT_PUBLIC_SITE_URL`) before falling back to forwarded headers, because Hostinger/Traefik server-action requests can surface localhost and Supabase will embed that wrong origin in the emailed callback URL.
- After activating a newly purchased Hostinger domain, reapply the VPS-facing `@` A record and `www` CNAME because the setup flow can reset the zone back to Hostinger parking IPs before the final DNS state propagates.
- In `CSVUploader`, when Papa Parse runs with `worker: true` + `step`, never rely on `complete(results.data)` being present; accumulate rows during `step` (with fallback guard) or uploads can fail with `Cannot read properties of undefined (reading 'data')` and show no progress.
- In connect-only runs, on selector exhaustion (`all_paths EXHAUSTED`) or send-button failure (`NOT_FOUND`/`CLICK_FAILED`), always capture a full-page screenshot and log its absolute path for deterministic UI failure triage.
- Follow-up sender runs must first requeue stale `followups.status='PROCESSING'` rows older than 45 minutes and must send only on direct message surfaces (`surface == "message"`), never on `connect_note`.
- NUDGE follow-up sends must resolve content from current sequence templates (`second_message`/`third_message`) using lead `sequence_step` when `attempt` is missing, and direct-message sanitization must never apply 300-char invite-note truncation.
- In `run_all.sh` under `set -u`, cleanup loops must use `"${TAIL_PIDS[@]-}"` and `"${SERVICE_PIDS[@]-}"` so `--help` or early exits do not fail with unbound PID arrays.
- Sequence authoring must reject any placeholder outside `{{first_name}}`, `{{last_name}}`, `{{full_name}}`, `{{company_name}}` across `{{...}}`, `{...}`, and `[...]` token classes in both `SequenceEditor` and `saveOutreachSequence`, while sender runtime keeps legacy aliases (including `{{VORNAME}}`/`{{NACHNAME}}`) for already-stored rows.
- Sequence variable pickers mounted near the bottom of long forms must render as anchored popovers with explicit z-index and upward placement (or viewport-aware placement), otherwise the dropdown opens below the fold and appears missing to users.
- Production auth must fail closed: if Supabase public auth env vars are missing, protected routes must redirect to `/login?e=config` and the login page must surface the misconfiguration instead of silently exposing Mission Control anonymously.
- Protected UI pages must call the server-side auth guard before rendering page data so the server, not only middleware, emits the login redirect when Mission Control is accessed anonymously.
- Hostinger must receive the Supabase runtime env on deploy (`NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`) or the entrypoint will deliberately fall back to `run_all.sh --web` and sequence saves will never hydrate the live list.
- If the browser console shows `Failed to load resource: 404` on `/favicon.ico`, do not treat it as a sequence-save failure; ship an explicit app icon and metadata icon link so the browser stops falling back to a missing favicon.
- Server-action saves that mutate lists rendered from server components must also refresh the client tree or optimistically update local state after success; `revalidatePath` alone does not make new sequences appear until a manual reload.
- LinkedIn credential saves and worker decrypts must resolve the same 32-byte key from `LINKEDIN_CREDENTIALS_KEY` or the shared fallback file `LINKEDIN_CREDENTIALS_KEY_FILE`/`.linkedin_credentials_key`; never surface "Credential encryption is not configured" unless both sources are missing or unreadable.
- Production Supabase magic links must use `NEXT_PUBLIC_SITE_URL` as the only callback origin; never fall back to forwarded request hosts in production, or the app can mint links for internal listeners like `0.0.0.0:3000` and loop on `otp_expired`.
- Connect-only invite failures on some profile layouts were timing-sensitive: after clicking `Mehr/More` or the invite action, wait longer for the modal and match both menuitems and buttons before declaring the path exhausted, because LinkedIn can render `Ohne Notiz senden` only after the dialog settles.
- Scraper launch routes must enforce a single active worker per `enrichment.pid`; if a pid is already alive, return 409 and refuse a second `/api/enrich` or `/api/enrich/connect-only` spawn so duplicate Playwright sessions do not race the same queue.
- Production LinkedIn browser workers must force headless Playwright on Linux when `DISPLAY`/`WAYLAND_DISPLAY` is absent; Docker/Hostinger containers have no GUI, so `headless=False` can spawn but never progress the queue.
- `connect_only` scraper runs must skip `enrich_one()` entirely and go straight to `send_connection_request()`; never reintroduce scrape-first behavior on the invite-only path.
